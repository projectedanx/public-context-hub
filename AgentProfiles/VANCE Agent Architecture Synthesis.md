## **4.1 JSON-RPC 2.0 Payload Topologies**

JSON-RPC 2.0 is a transport-agnostic, lightweight, stateless remote procedure call protocol utilizing the JSON data format (RFC 4627).14 The specification defines absolute structural requirements for interaction.

**Request Objects:** When the IDE client queries the VANCE server, it transmits a Request object containing the following members 14:

* jsonrpc: A String specifying the version. It MUST be exactly "2.0".14  
* method: A String declaring the requested operation (e.g., textDocument/definition). Methods beginning with rpc. are reserved for internal protocol extensions.14  
* params: An OPTIONAL structured value. If provided "by-position", it MUST be an Array. If provided "by-name", it MUST be an Object, and all member name matching is strictly case-sensitive.14  
* id: An identifier established by the client, which MUST be a String, Number, or Null.14 Fractional numbers are strongly discouraged due to binary representation issues, and the use of Null in a standard request is highly discouraged to prevent confusion with JSON-RPC 1.0 architecture.14

If a Request object completely omits the id member, it is classified as a **Notification**.14 Notifications indicate that the client has no interest in a response. Consequently, the server MUST NOT reply to a Notification, meaning the client remains entirely unaware of any internal errors (such as "Invalid params" or "Internal error") that the notification might trigger within the VANCE ingestion engine.14

**Response Objects and Error Matrices:** For every standard Request, the VANCE server MUST emit a Response object.14 The protocol strictly dictates that a Response MUST contain either a result member (on success) or an error member (on failure), but it MUST NEVER contain both simultaneously.14 The id member is REQUIRED and must flawlessly match the id of the originating Request.14

When the VANCE server encounters a structural anomaly, it must project an Error object containing a code (Integer), a message (String), and optional data.14 The protocol reserves specific integer ranges for pre-defined errors, which VANCE maps to specific epistemic states:

| Error Code | Designation | Trigger Condition within VANCE Architecture |

| **\-32700** | Parse error | The client payload is fundamentally malformed or fails Tree-Sitter ingestion.14 |
| **\-32600** | Invalid Request | The payload lacks the rigid jsonrpc: "2.0" protocol invariant.14 |
| **\-32601** | Method not found | The client attempts to invoke an unsupported LSP capability not yet mapped by VANCE.14 |
| **\-32602** | Invalid params | A structural deviation detected in the by-position or by-name parameter mapping.14 |
| **\-32603** | Internal error | The agent's CFDI circuit breaker triggers during the \+++SilentReasoning escrow phase, forcing a halt.14 |

The protocol also supports **Batch Requests**, allowing the client to send an Array of Request objects simultaneously.14 The VANCE server processes these concurrently, returning an Array of Response objects in any arbitrary order.14 If a batch consists solely of Notifications, the server MUST return nothing at all, not even an empty array.14 This requires VANCE to maintain exceptional state isolation, ensuring concurrent semantic queries do not bleed across request contexts.
