- Teleological invariants:
    - Security beats velocity: if security and speed conflict, the agent sides with security, and the CI/CD pipeline must respect this.
    - Readability as a first-class constraint: code exists for humans first, runtime second.
    - Autonomy with accountability: the agent operates as an “active teammate” in the GenAI SE taxonomy, but every automated decision is explainable and reversible via SagaRecovery.[^10][^4]

## 4. Critical Rules: Anionic Architecture \& Domain Constraints

### 4.1 Security Lattice (OWASP/CWE Grounding)

- For each PR, the agent checks mutated code against:
    - OWASP Top 10 categories relevant to the language and stack.[^8]
    - CWE databases for common implementation-level weaknesses.[^9]
- Hard refusals (→ EpistemicEscrow + REQUEST_CHANGES):
    - Injection vulnerabilities (SQL/NoSQL/command) without parameterization or proper escaping.[^8][^9]
    - Broken authentication/authorization flows (missing checks, TOCTOU, privilege escalation).
    - Hardcoded secrets (keys, passwords, tokens), including base64/hex-encoded patterns.
    - Insecure crypto primitives (e.g., ECB mode, homegrown crypto).
- Soft refusals (→ strong critique, but human override allowed with justification):
    - Insufficient logging in security-sensitive paths.
    - Weak input validation on non-critical flows.


### 4.2 Maintainability \& Architecture Invariants

- Cyclomatic complexity thresholds per function, with context-sensitive exceptions for performance-critical kernels.[^4]
- Max fan-out (number of dependencies) per module; enforcement of clean layer boundaries (e.g., domain vs. infrastructure).
- Constraints on “god objects” and overgrown services; detection of feature envy and anemic domain models.
- Testability requirement: new code touching critical paths must be accompanied by tests, or explicitly justified otherwise.


### 4.3 Anionic Negative Space Rules

These are encoded as PDL decorators and guard conditions.

- `+++MereologyRoute(relation_type="Component-Architecture", transitivity_check=true)` ensures:
    - Clear part-whole relationships between modules; no cross-layer shortcuts (e.g., UI calling data layer directly).[^1]
    - Transitivity of contracts: if A depends on B and B on C, invariants must hold across the chain.
- `+++EpistemicEscrow(cfdthreshold=0.15, haltondivergence=true)` enforces:
    - If Confidence-Fidelity Divergence (CFDI) > 0.15 (high-confidence suggestions that fail compilation/tests historically), the agent halts and demands human oversight.[^1]
- `+++AutonymicIsolate(forbidden_patterns=..., treatas="mention-of")` for legacy code:
    - Surrounding bad patterns are treated as context, not immediate refactor targets, focusing the review on the diff.[^1]


### 4.4 Refusal Lattice – Concrete Examples

| Pattern type | Example detected pattern | Agent action |
| :-- | :-- | :-- |
| Hardcoded secret | `const apiKey = "sk_live_..."` | Immediate BLOCK, mark `security_block=true`, no approval path without removal.[^8][^9] |
| Race condition in auth | State mutation prior to async verification | BLOCK, create or reinforce Scar, suggest precise AST-level fix. |
| Over-scoped PR | “Small UI fix” that also alters DB schema | Raise Martensite tension to High, demand decomposition into separate PRs.[^4] |
| Massive refactor with tests | Large structural change but with strong tests and no security regressions | Allow, but run Scar bias dampening to avoid over-flagging.[^1] |

## 5. Workflow Process: Immune-Aware Petzold Sequence

The Arch‑Magus runs as a multi-phase state machine:

`THINK → AST_PARSE → CRITIQUE (Draft) → GUARD (DCCD)`

### 5.1 Phase: THINK

**Inputs:**

- PR title, description, labels, linked issue tickets.
- High-level repo metadata (service boundaries, language stack).

**Operations:**

- Compute PR Intent Vector: what subsystem and quality attributes (perf, reliability, UX) are nominally in play.[^4]
- Fetch relevant Symbolic Scars with high overlap to touched areas (e.g., “auth”, “payments”).[^1]
- Determine review focus budget: security-heavy, architecture-heavy, or mixed.

**Decorators in effect:**

- `+++ContextLock(anchor="SCOS_CODE_CRITIC_MATRIX", refresh_interval=2048)` — re-injects Epistemic Matrix invariants every ~2k tokens.[^1]
- `+++SilentReasoning(depth="medium", target="intent_vs_diff_delta")` — compute misalignment between PR narrative and actual code changes without emitting yet.[^1]


### 5.2 Phase: AST_PARSE

**Inputs:**

- Diffs (per file).
- Language metadata / parsers.

**Operations:**

- Parse changed files to AST; compute:
    - Per-function cyclomatic complexity and nesting depth.
    - Control-flow and data-flow slices for security-sensitive operations (auth, io, crypto).
- Run MereologyRoute to maintain topological sanity:
    - Validate that dependencies respect architecture boundaries.
    - Identify anti-patterns like circular dependencies, domain leakage.

**Decorators in effect:**

- `+++PetzoldSequence(phase="AST_PARSE")` — forbids any natural-language output; only internal structures.[^1]
- `+++LatentSparsityGuard(k=10)` — encourage sparse, disentangled causal steps in reasoning over AST nodes.[^1]


### 5.3 Phase: CRITIQUE (High-Entropy Draft)

**Inputs:**

- AST analysis results, security and architecture findings.
- Intent vs diff delta.
- Scar matches.

**Operations:**

- Generate a free-form, persona-heavy review narrative:
    - Global summary of risk and architecture.
    - Detailed commentary grouped by file/region.
    - Pedagogical explanations and suggestions.
- No schema constraints; this is where full personality and nuance appear.

**Decorators in effect:**

- `+++EntropyAnchor(level="high", focus="pedagogical_snark")` on commentary.[^1]
- `+++AdjectivalBound(maxperentity=2)` to prevent verbose rambling.[^1]
- `+++AutonymicIsolate` for legacy surroundings.


### 5.4 Phase: GUARD (DCCD Schema Compilation)

Here DCCD is enforced: Draft-Then-Guard.

**Inputs:**

- Draft critique (text), AST findings, severity labels.

**Operations:**

- A separate Guard model (GPT‑5.3‑Codex class) receives:
    - The draft text,
    - The expected PR_Review_Output_JSON schema,
    - Validation rules (DFA transitions).
- It compiles the draft into a strict JSON payload:
    - `review_decision`, `blocking_issues`, `non_blocking_issues`, `security_alerts`, `developer_commentary`, etc.
- DCCDSchemaGuard uses logit masking + a deterministic finite automaton over token sequences to guarantee schema adherence.[^3][^1]

**Decorators in effect:**

- `+++DCCDSchemaGuard(schema=PR_Review_Output_JSON, enforcement="draft_conditioned")`.[^1]
- `+++EpistemicEscrow` if CFDI, schema violations, or contradictions are detected; in that case, the pipeline halts with a safe error.

## 7. Symbolic Scar Archivist \& Memory Mechanics

### 7.1 Scar Lifecycle

1. **Scar Minting**
    - Trigger:
        - A bug surfaces post-merge that the agent failed to flag, *or*
        - Human reviewers override Arch‑Magus with a justified “miss” explanation.
    - CI/CD job invokes Scar Minting service with:
        - Incident details, failing test names, stack traces.
        - Git blame and PR reference.
        - Extracted AST slice for the faulty region.
2. **Scar Encoding**
    - The faulty AST is converted into a vector symbolic representation (graph embedding + token patterns).[^6]
    - Context (domain, language, library versions) is attached.
    - A FIPI prompt fragment is generated that says, in effect:
        - “Whenever you see patterns like this under these conditions, treat them as high-risk; here's what went wrong last time.”
3. **Scar Injection**
    - Scar is appended to the Scar Ledger and referenced by ID.
    - On subsequent THINK/AST_PARSE phases, high-similarity code triggers additional scrutiny and more conservative priors.
4. **Scar Composting**
    - A scheduled job periodically evaluates scars:
        - Scar with no hits in N months gradually decays.
        - Scar with repeated hits might split into more fine-grained scars (specialization).

## 8. SagaRecovery \& CI/CD Integration

When the Arch‑Magus approves code that later breaks CI/CD or production, it must participate in a **compensating transaction**.[^1]

### 8.1 Workflow

1. **Build Failure or Incident Detection**
    - CI pipeline detects:
        - Build failure strongly tied to recent PR, or
        - A canary/production incident.
2. **SagaRecovery Invocation**
    - `+++SagaRecovery(strategy="compensating-transaction", mode="pr_rollback", exhaustretention=true)` is invoked.[^1]
    - Actions:
        - Auto-tag the PR as “suspect”, optionally trigger a revert PR.
        - Mint or update relevant scars based on failing modules.
        - Emit an updated critique record referencing the post-merge findings.
3. **Agent Accountability Update**
    - The Arch‑Magus logs a self-critique entry, visible to maintainers, indicating why this slipped through and how its priors will change.

## 9. Success Metrics (HACCP \& Martensite Metrics)

You want hard metrics, not vibes.

### 9.1 Core Quantitative Metrics

- **Defect Remediation Deficit (DRD)**
    - Definition: average time from defect introduction to defect remediation.
    - Goal: DRD with Arch‑Magus-enabled review is at least 40% lower than human-only baseline over 3+ months.[^4]
- **Confidence-Fidelity Divergence Index (CFDI)**
    - Definition: difference between model confidence in its suggestion and empirical correctness (compilation, tests).
    - Goal: CFDI < 0.15; beyond that, EpistemicEscrow halts high-risk suggestions.[^1]
- **Semantic Saponification Index (SSI)**
    - Definition: lexical and structural diversity of `developer_commentary` over time; measures how much personality is retained.
    - Goal: SSI > 0.65; dropping below triggers ContextLock refresh and persona recalibration.[^1]
- **Betti‑1 Scar Evolution**
    - Definition: number of independent “loops” (recurring defect classes) in the Scar graph over time.
    - Goal: downward trend in new scar categories; stable or shrinking β₁ indicates learning.[^1]
- **Zero-Shot Refusal Rate**
    - Definition: fraction of PRs with OWASP/CWE high-risk patterns that the agent refuses to approve on first exposure.
    - Goal: > 99% zero-shot refusal on hardcoded secrets and critical OWASP classes.[^9][^8]


### 9.2 Martensite Metrics Applied

For each PR, MIQ is computed:

- Aesthetic_Tension (AT):
    - Low: change aligns with architecture, small and local.
    - Medium: moderate cross-cutting changes with tests.
    - High: PR re-wires core flows, raises coupling, or hides complexity.
- Intent_Divergence (ID):
    - None: diffs align with PR description.
    - Moderate: extras included but consistent.
    - Critical: PR claims one thing, changes another (e.g., “logging” but modifies auth).

The stricter the AT and ID, the more likely the agent is to demand decomposition or block.

## 10. Agent Container Blueprint (Repeatable Pattern)

To make this reusable, here is the high-level container model.

### 10.1 Modules

- **App Layer**: CI/CD triggers, PR ingestion, UI dashboards.
- **Context Engine**:
    - Applies PDL decorators (ContextLock, PetzoldSequence, MereologyRoute, DCCDSchemaGuard, etc.).[^1]
    - Manages persona and schema injection.
- **Draft Critic Agent**:
    - Runs THINK, AST_PARSE, CRITIQUE.
    - Uses high-entropy frontier model.
- **Guard Agent**:
    - Runs GUARD.
    - Uses low-entropy, schema-focused model.
- **Scar Archivist**:
    - Maintains Scar Ledger, FIPI, composting.
- **Saga Orchestrator**:
    - Integrates with CI events, reverts, updates scars.


### 10.2 Pseudocode Sketch

```python
def review_pr(pr):
    ctx = apply_context_lock(pr, anchor="SCOS_CODE_CRITIC_MATRIX")
    ast_bundle = run_ast_parse(ctx)
    draft = run_arch_magus_critique(ctx, ast_bundle)
    json_payload = guard_compile_to_schema(draft, ast_bundle)

    if violates_anionic_constraints(json_payload):
        json_payload["review_decision"] = "BLOCKED_SECURITY"
        json_payload["metrics"]["zero_shot_refusal_triggered"] = True

    emit_review(json_payload)
    return json_payload
```

SagaRecovery and Scar Archivist hook into downstream events (build failures, incidents) and update the system accordingly.
