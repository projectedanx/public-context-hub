## WSL2: Secure Foundations for Agentic AI Development

This guide provides a comprehensive framework for building a **secure, performant, and governable** environment for developing AI agents using **WSL2 (Windows Subsystem for Linux 2)**.

### **I. Foundational WSL2 Architecture & Setup**

#### **Core Principles**
1.  **Optimization Cardinal Rule:**  
    **✅ STORE ALL PROJECT FILES INSIDE THE LINUX FILESYSTEM (`\\wsl$\`, `~/`)**  
    **🚫 NEVER WORK DIRECTLY ON `/mnt/c` (Windows files)**  
    *Why?* Accessing `/mnt/c` (DrvFS) via 9P protocol is **20x slower** than native Linux ext4. Critical for AI workloads (large datasets, model files, code generation).

2.  **WSL2 vs. WSL1:**
    *   **WSL2:** Full Linux kernel in lightweight VM. **Required for:** Docker, systemd, advanced networking, true Linux performance. *Best for AI Dev.*
    *   **WSL1:** Translation layer (no VM). **Use only for:** Simple command-line tools, legacy apps incompatible with WSL2.

#### **Essential Configuration Files**

| **File**          | **Scope**       | **Key Settings**                                                                 | **Purpose**                                                                                                                                 |
|-------------------|-----------------|---------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------|
| **`.wslconfig`**  | **Global VM**   | `memory=<size>`, `processors=<num>`, `localhostForwarding=true`,<br>`networkingMode=mirrored`,<br>`[boot] systemd=true`<br>`[automount] enabled=true, options="metadata"`<br>`[interop] appendWindowsPath=false` | Tunes the **entire WSL2 VM**. Critical for performance, resource limits, networking, and security. `appendWindowsPath=false` **prevents Windows PATH pollution** (critical for security). |
| **`wsl.conf`**    | **Per-Distro** | `[automount] enabled=true, options="metadata", mountFsTab=true`<br>`[network] generateResolvConf=false`<br>`[user] default=<user>` | Customizes **individual Linux distros**. `metadata` ensures proper permissions on mounted drives. Disable resolv.conf generation if using custom DNS. |

#### **Installation & Management Commands**
```bash

# List installed distros
wsl --list --verbose
```

### **II. Security & Secrets Management (WSL2 Posture)**

#### **Critical Risks & Mitigations**
| **Risk**                          | **Mitigation**                                                                                                                               |
|-----------------------------------|----------------------------------------------------------------------------------------------------------------------------------------------|
| **Secrets on `/mnt/c`**           | **NEVER store secrets on Windows (`/mnt/c`).** Use Linux filesystem (`~/.secrets`, project `secrets/`) with strict permissions.              |
| **`wsl.exe -u root` Bypass**      | Avoid running commands as root via `wsl.exe -u root`. Use `sudo` inside WSL normally.                                                        |
| **`WSLENV` Leakage**              | **NEVER use `WSLENV` to pass secrets.** It's insecure and exposes variables to Windows.                                                       |
| **Windows Malware Access**        | Secrets on `/mnt/c` are accessible to Windows malware. **Keep secrets ONLY in Linux FS.**                                                 |
| **PATH Pollution**                | `appendWindowsPath=false` in `.wslconfig` prevents Windows executables from accidentally overriding Linux tools (e.g., `python`, `node`). |

#### **Local Secrets Management Best Practices**
1.  **`.env` Files:**
    *   Store in project root or `~/.secrets/`.
    *   **ALWAYS** add `.env` to `.gitignore`.
    *   Provide `.env.example` with placeholders.
    ```bash
    # .gitignore
    .env
    secrets/
    ```
    ```env

    OPENAI_API_KEY=your_key_here
    GEMINI_API_KEY=your_key_here
    ```

2.  **`direnv` (Recommended):**
    *   Scopes environment variables **per-directory**.
    *   Auto-loads `.envrc` when entering a directory.
    *   **Ephemeral** – doesn't leak to other sessions.
    ```bash
    # Install direnv
    sudo apt install direnv
    # Enable for bash/zsh
    echo 'eval "$(direnv hook bash)"' >> ~/.bashrc
    # Create .envrc in project root
    echo 'export OPENAI_API_KEY="sk-..."' > .envrc
    direnv allow . # Approve the .envrc
    ```

    *   **High:** Require mandatory human approval (e.g., DB schema changes)

### **V. Advanced Multi-Agent Collaboration**

#### **DRP-MULTI-AGENT-PROTOCOL-2025**
A standardized protocol for agent handoffs.

1.  **Agent Role Taxonomy:**
    *   **ArchitectAgent:** Decomposes requirements.
    *   **PlannerAgent:** Sequences tasks & dependencies.
    *   **CoderAgent:** Generates/modifies code.
    *   **ValidatorAgent:** Runs tests, static analysis, security scans.
    *   **UserProxyAgent:** Manages ambiguity resolution (HITL).

2.  **Handoff Protocol:**
    *   Agents communicate via **`agent_packet.json`**.
    ```json
    {
      "protocol": "DRP-2025",
      "from": "CoderAgent",
      "to": "ValidatorAgent",
      "task_id": "task_123",
      "payload": {
        "file": "src/utils.py",
        "content": "...",
        "reasoning_trace": "..." 
      },
      "checksum": "sha256:..."
    }
    ```

3.  **Reflexive Logic:**
    *   `ValidatorAgent` routes failures **upstream** to `CoderAgent` or `PlannerAgent`.

#### **Explainable AI Codeflow Framework**
*   **Core Pattern:** `CoderAgent -> ExplainerAgent -> ValidatorAgent`
*   **Explanation Packet (JSON Schema):**
    ```json
    {
      "task_id": "task_123",
      "generated_code": "...",
      "reasoning": {
        "step_1": "User needs auth endpoint",
        "step_2": "Choosing JWT over OAuth2 for simplicity",
        "step_3": "Generated /auth/login endpoint"
      },
      "alternatives_considered": ["OAuth2", "Session cookies"],
      "confidence": 0.92
    }
    ```
*   **Artifacts:**
    *   Embedded comments in generated code.
    *   `reasoning_diff.md` tracking logic drift over versions.

#### **Symbolic Cohabitation Workspace**
*   **Workspace Specification (`workspace.yaml`):**  
    Defines agent roles, memory scopes, and permissions.
    ```yaml
    agents:
      coder:
        role: CoderAgent
        memory: shared_bus + local_shard
        permissions: [write_code, read_specs]
      validator:
        role: ValidatorAgent
        memory: shared_bus
        permissions: [run_tests, approve_code]
    ```
*   **Epistemic Memory Engine:**
    *   **Shared Bus:** Global project context (GEMINI.md, prompt registry).
    *   **Agent-Local Shards:** Private working memory per agent.
*   **Portability:** Export state as `.collabpkg.zip` for replayable collaboration.

### **VI. Optimization & Performance**

*   **Faster File I/O:** Native Linux ext4 filesystem (`~/`) is **20x faster** than `/mnt/c`.
*   **Avoid Docker in WSL2:** WSL2 lacks full syscall compatibility. Use Linux host or Docker Desktop instead.
*   **Interoperability:** Use `wslpath` to safely convert Windows paths to WSL paths.
  ```bash
  wslpath C:\Projects\MyApp
  # Output: /mnt/c/Projects/MyApp
  ```
