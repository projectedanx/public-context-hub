## Refactoring as Infrastructure: Patterns, Risk Audits \& Evolutionary Guarantees in Software Systems

### Executive Summary

This Design Requirements Proposal (DRP) codifies refactoring from a craft practice into a systematic, measurable infrastructure capability. Refactoring—the controlled restructuring of code without changing observable behavior—has evolved from an optional code hygiene practice to a critical operational discipline. This framework enables organizations to safely transform legacy and production systems by combining three foundational pillars: (1) **empirically-validated refactoring patterns** grounded in 724 real-world pull requests, (2) **formal risk diagnosis** that detects semantic dangers before execution, and (3) **measurable safety guarantees** combining static analysis, behavioral contracts, and evolutionary regression testing.

The framework integrates emerging AI-assisted refactoring capabilities (Large Language Models) with battle-tested safeguards, acknowledging that raw LLM output carries a 6-7% unsafe refactoring rate that must be mitigated through detect-and-reapply validation pipelines. By instrumenting refactoring as infrastructure—rather than treating it as ad-hoc maintenance—teams reduce regression risk by up to 100%, improve technical debt visibility, and enable continuous refactoring as a competitive advantage.

***

### I. Problem Context \& Stakeholder Value

**The Refactoring Imperative**

Codebases ossify without systematic refactoring. As feature accumulation proceeds, three failure modes emerge:

1. **Technical Debt Compounding**: Each non-refactored architectural violation multiplies downstream development cost. SQALE measurements show teams allocating 50% of effort to maintenance rather than feature development. Unaddressed debt becomes "interest payment" in form of slower delivery velocity, higher defect rates, and team churn.[^1]
2. **Silent Regression Risk**: Manual refactoring without formal validation introduces behavioral changes in 6-7% of cases. These semantic bugs propagate to production if test coverage gaps exist, costing far more than preventive validation would.[^2]
3. **Knowledge Decay \& Team Lock-in**: Legacy systems without clear refactoring provenance become difficult to understand. Departing developers take implicit design knowledge; new team members lack confidence to refactor. This creates organizational bottlenecks around code changes.

**Stakeholder Decisions Enabled**

- **Engineering Leads**: Cost/benefit analysis of refactoring vs. feature work; prioritization by risk-adjusted impact
- **Developers**: When/how to apply specific patterns; confidence that refactors won't break production
- **DevOps/SRE**: Risk-aware release gates; measurement of code quality improvement velocity
- **Security Auditors**: Auditable trails of refactoring changes; elimination of implicit code path modifications

### III. Risk Audit Framework: The ReFD Model

**The Problem with Traditional Safeguards**

Precondition-based refactoring engines (Eclipse, IntelliJ) prevent unsafe refactorings through overly conservative rules:

- **Overly weak**: Allow refactorings that will fail (e.g., moving methods with inaccessible field references)
- **Overly strong**: Reject legitimate refactorings without explanation (e.g., move method when superclass override exists—solvable via wrapper method)
- **No diagnostics**: Users receive "refactoring rejected" with no insight into which aspect failed or how to remediate

Testing-based validation catches bugs *post-hoc* and suffers incomplete coverage gaps. Formal proofs are infeasible for complex languages like Java.[^4]

**The ReFD Risk Hierarchy: Graduated Detection Before Execution**

ReFD (Refactoring Diagnosis Framework) introduces a graduated semantic risk model executed *before* refactoring:[^4]

```
Microstep → Potential Risk → Actual Risk → Danger → Error
           (universal to     (specific to  (unmitigated  (runtime/
            this operation)   this code)    across full   compile
                                           sequence)     time)
```

**Definitions**

1. **Microstep** (atomic code modification)
    - Examples: AddMethod, RemoveMethod, MoveMethod, RenameClass
    - Each carries multiple potential risks regardless of context
2. **Potential Risk** (risk inherent to the microstep type)
    - AddMethod: Double definition (method already exists)
    - RemoveMethod: Missing definition (caller references deleted method)
    - MoveMethod: Broken local reference (field becomes inaccessible)
    - **Not** context-dependent; always theoretically possible
3. **Actual Risk** (potential risk confirmed in target code context)
    - Detector queries code context (inheritance hierarchy, call sites, field scope)
    - Returns set of program locations where risk manifests
    - Example: "RemoveMethod m" → Detector finds 7 call sites of m in codebase → Actual Risk ✓
4. **Danger** (actual risk *not* mitigated by subsequent microsteps)
    - Verdict mechanism filters false positives
    - Example: AddMethod m, then RemoveMethod m → initial double-definition risk is mitigated → Danger ✗
    - Only unmitigated actual risks become dangers
5. **Error** (behavioral change manifested at runtime or compilation)
    - May or may not result from detected dangers
    - ReFD detects dangers (which *could* cause errors) not errors themselves

**Key Dangers by Operation Type**


| Microstep | Key Dangers | Detector Logic |
| :-- | :-- | :-- |
| **AddMethod(m, C)** | AM-1: Double definition if `C.superclass has m(sig)` or `C.subclass has m(sig)` with different semantics | Query inheritance hierarchy for signature + semantic conflicts |
|  | AM-2: Breaks Liskov Substitution Principle | Check if override violates contract |
|  | AM-3: Overload ambiguity with parameter type conversion | Check signature collisions with type coercion |
| **RemoveMethod(m, C)** | RM-1: Missing definition (unreachable callers) | Data-flow analysis + call-site tracking |
|  | RM-2: Lost concrete implementation if C.m overrides C.superclass.m | Query override relation in AST |
|  | RM-3: Lost semantic specification if m is interface contract | Check interface/abstract method declarations |
| **MoveMethod(m, C→D)** | MM-1: Broken local references (C.fields become inaccessible in D) | Static scope analysis: C.fields not in D scope |
|  | MM-2: Broken inherited method calls | Query which methods C.m calls that depend on C context |
|  | MM-3: Polymorphic call path changes | Dynamic binding analysis for v.m() calls |
| **RenameEntity(e, old→new)** | RE-1: Unintended override in subclass | If renaming C.m to D and subclass has m(sig), dynamic dispatch now calls subclass |
|  | RE-2: Broken external references (cross-module) | Beyond single-file scope (acknowledged limitation) |

**Example: Move Method Refactoring**

*Code before:*

```java
class Source {
  private int local = 15;
  public void method(Target target) {
    target.doSomething();
    System.out.println("Executing with " + local);
  }
}
class Target {
  public void doSomething() { ... }
}
```

*Intended after:*

```java
class Target {
  public void method(Source source) {
    doSomething();
    System.out.println("Executing with " + source.getLocal());
  }
}
```

**ReFD Risk Diagnosis:**

- Microstep 1: AddMethod(Target.method)
    - Potential Risk: Double definition
    - Actual Risk? Check if Target already has method(Source) → No, actual risk = ✗
    - Potential Risk: Breaks subtyping (if Target has subclass Sub with method(Source))
    - Actual Risk? If Sub exists with method(Source), actual risk = ✓ → **Danger: Unintended override**
    - *Solution:* Rename to methodFromSource() or add @VisibleForTesting annotation
- Microstep 2: Remove Source.method
    - Potential Risk: Missing definition
    - Actual Risk? Find all call sites of Source.method → 3 found in codebase, actual risk = ✓
    - *Solutions:* Redirect to Target.method via wrapper, or update all callers
- Microstep 3: Update all call sites
    - Verdict: Previous actual risks mitigated by systematic updates → Danger ✗

**ReFD Tool Implementation**

```
Program Graph (AST + dataflow) ← Atlas adapter
        ↓
Subdetector Library (reusable queries on program locations)
        ↓
Detector Library (composes subdetectors for each potential risk)
        ↓
Refactoring Declaration (composites of microsteps with detectors)
        ↓
Verdict Mechanism (refactoring-specific filtering of false positives)
        ↓
Risk Report (exact locations + remediation suggestions)
```

ReFD is implemented as Eclipse plugin (GitHub: NHLStenden-ISAL/ReFD); enables visual navigation of detected dangers with code-location highlighting and suggested fixes.

### IV. Metrics \& Behavioral Validation

**Cyclomatic Complexity \& CRAP Metric**

Cyclomatic complexity (CC) measures independent paths through code. High complexity correlates with refactoring necessity and test coverage inadequacy.[^5][^6]


| CC Range | Risk Level | Refactoring Action | Test Coverage Target |
| :-- | :-- | :-- | :-- |
| 1–10 | Low | Optional decomposition | None required |
| 11–20 | Moderate | Decompose into sub-methods | 50% minimum |
| 21–50 | High | Extract class + interface | 80% minimum |
| >50 | Untestable | Mandatory redesign + break dependencies | Impossible to achieve 100% |

**CRAP Index** (Change Risk Anti-Pattern) = CC² × (1 - coverage%)³ / 100

- Threshold: CRAP ≥ 30 indicates risk zone
- CC=10 with 42% coverage → CRAP ≈ 30 (marginal)
- CC=25 with 80% coverage → CRAP ≈ 24 (acceptable, but requires sustained test investment)
- CC>30 → No amount of testing can mitigate; requires structural refactoring

**SQALE Technical Debt Framework**[^7][^1]

SQALE (Software Quality Assessment based on Lifecycle Expectations) provides financial transparency for refactoring investment:

**Quality Model** = Organization's definition of "right code"

- Each violation of quality standard creates a debt item
- Debt items are detected by static analysis tools (SonarQube, CodeAnt.ai)

**Remediation Cost** = Time-to-fix for each violation type

- Critical security flaw: 4 hours
- Code duplication: 1 hour
- Naming violation: 15 minutes
- Violations × remediation time = principal + interest

**SQALE Rating** (A–E grading based on debt density)

- Debt Density = Total Remediation Cost / Lines of Code
- A grade: <5% → manageable debt
- E grade: >50% → requires immediate intervention

**Portfolio-Level Debt Visualization**
Each application positioned on matrix:

- X-axis: Debt density (technical overhead)
- Y-axis: Non-remediation cost (business impact)
→ Quadrant analysis reveals risk/investment tradeoff

| :-- | :-- | :-- | :-- |

### V. LLM-Assisted Refactoring \& Safety Validation

**The LLM Opportunity \& Risk**

Large Language Models (LLMs like GPT-4, Gemini) bring unprecedented capability to refactoring:

- Automated opportunity identification (from code description to refactoring recommendations)
- Natural language reasoning about intent and design
- Fast prototyping of alternative designs

**However, raw LLM refactoring is unsafe**: 7.4% of GPT-4 and 6.6% of Gemini refactorings introduce semantic bugs or syntax errors.[^2]

**Empirical LLM Refactoring Performance**[^2]

**Opportunity Identification (RQ1)**


| Prompt Type | GPT Success | Gemini Success | Key Finding |
| :-- | :-- | :-- | :-- |
| Generic (P1): "Refactor this code" | 15.6% (28/180) | 3.9% (7/180) | Without guidance, LLMs miss 84% of opportunities |
| Type-Specific (P2): "Apply extract method to this code" | 52.2% (94/180) | 21.1% (38/180) | **+236% improvement (GPT)**, +442% (Gemini) |
| P2 + Explicit Motivation (P3): "Extract method to reduce duplication of getConfig()" | 86.7% (156/180) | 68.3% (123/180) | Subcategory + search space restriction is critical |

**Solution Quality (RQ2)**

Out of 180 validated refactorings:

- GPT: 176 recommendations → 63.6% ≥ human expert quality, 31.5% poor, 4.9% buggy
- Gemini: 137 recommendations → 60.6% ≥ expert, 33.6% poor, 5.8% buggy

**Bugs in LLM Refactorings**[^2]


| Type | Count | Example |
| :-- | :-- | :-- |
| Semantic (functionality change) | 18 | Replaced null-check `if (x != null)` with unsafe ternary `x != null ? doX() : doY()` causing NullPointerException |
| Syntax (compile error) | 4 | Extracted variable into catch-block scope where it's inaccessible |
| **Total Unsafe** | **22/313 (7.0%)** |  |

**Solution: RefactoringMirror (Detect-and-Reapply)**[^2]

```
Original Code (c) →[LLM]→ LLM Output (c') 
  ↓
[ReExtractor] Identify refactorings between c and c'
  ↓
[Filter] Extract only refactoring operations, ignore buggy optimizations
  ↓
[IntelliJ IDEA Refactoring Engine] Reapply identified refactorings to c
  ↓
Safe Code (ĉ) with guaranteed behavior preservation
```

**RefactoringMirror Results**

- Refactorings correctly reapplied: 94.3% (33/35)
- Bugs avoided: 100% (22/22 unsafe suggestions filtered out)
- Quality preserved: 63.6% still ≥ expert level (LLM intelligence preserved, safety added)

**Why This Works**:

1. LLMs are good at *identifying what should be refactored* and *why*
2. LLMs are poor at *guaranteeing semantic equivalence* in code transformation
3. Battle-tested engines (IntelliJ, Roslyn) excel at behavior-preserving automation
4. Detect-and-reapply bridges the gap: leverage LLM reasoning + deterministic execution

### VII. Risk Assessment \& Mitigation Strategy

**Risk Taxonomy for Refactoring**


| Risk | Severity | Detection Method | Mitigation |
| :-- | :-- | :-- | :-- |
| Behavioral Change (silent regression) | CRITICAL | Regression testing + behavioral contracts | Test coverage ≥75%, ReFD diagnosis |
| External API Break (cross-module impact) | CRITICAL | Static analysis + cross-file dependency tracking | Semgrep rules for signature matching |
| Performance Degradation | HIGH | Benchmarking suite (run pre/post refactor) | Performance test gates in CI |
| Dead Code Introduction | MEDIUM | Static analysis (unused variables, methods) | Linting rules enforce cleanup |
| Knowledge Decay (poor documentation) | MEDIUM | Refactoring changelog + @refactor: annotations | Mandatory commit message format |
| Developer Uncertainty (fear of refactoring) | MEDIUM | Confidence metrics in test suite | >80% test coverage, clear git history |

**Risk Budgeting: Change Impact Heatmap**

Prioritize refactoring by risk-adjusted value:

```
High-Risk Modules (auth, billing, payments):
  - Change frequency (churn): Varies
  - Test coverage: Often <60% (legacy code)
  - Cyclomatic complexity: Typically 25-50
  - Risk score: (churn × complexity) / coverage
  → Refactor conservatively; require double-audit + canary release

Medium-Risk Modules (data transformation, utilities):
  - Churn: Moderate
  - Coverage: 60-75%
  - Complexity: 10-25
  → Standard refactoring; regression test gate

Low-Risk Modules (helpers, logging, formatting):
  - Churn: Low (stable API)
  - Coverage: Often >80%
  - Complexity: <10
  → Aggressive refactoring; encourage architectural cleanup
```

### VIII. Governance \& Execution Framework

**The Refactoring Lifecycle**

```
Phase 1: IDENTIFICATION
  ├─ Static analysis (SonarQube, CodeAnt.ai)
  ├─ Refactoring opportunity mining (RefactoringMiner)
  ├─ Team discussion (code review, architecture sync)
  └─ Prioritization by risk + value

Phase 2: DESIGN & VALIDATION
  ├─ Apply ReFD danger analysis (identify actual risks)
  ├─ Establish baseline metrics (complexity, coverage)
  ├─ Document refactoring rationale (@refactor: comment)
  ├─ Define success criteria (Δ metrics, test results)
  └─ Assign reviewer (senior developer for high-risk)

Phase 3: EXECUTION
  ├─ Create isolated feature branch
  ├─ Apply refactoring (manual or tool-assisted)
  ├─ Commit with tagged message (REFACTOR: Extract Method in PaymentProcessor)
  ├─ Bind to tracking system (Jira, Linear)
  └─ Push to CI for automated validation

Phase 4: VALIDATION
  ├─ Unit test suite passes (0 failures)
  ├─ Integration tests pass (0 failures)
  ├─ Regression test suite passes
  ├─ Code coverage Δ ≥ 0 (never decrease)
  ├─ Cyclomatic complexity Δ ≤ -10% (confirm simplification)
  ├─ Performance benchmark ±5% (no regression)
  └─ Security scan (no new vulnerabilities)

Phase 5: REVIEW & MERGE
  ├─ Diff auditability: Semantic delta must match documented rationale
  ├─ Peer review (focus on behavior preservation)
  ├─ Approval by code owner (2/3 for high-risk modules)
  └─ Merge and observe CI metrics

Phase 6: MONITORING
  ├─ Track velocity (time to next feature, lower is better)
  ├─ Defect rate post-release (should remain stable)
  ├─ Developer satisfaction (survey on codebase clarity)
  └─ Update refactoring log (refactor-log.json)
```

**Enforcement: CI/CD Gates**

```yaml
# Example: .github/workflows/refactoring-gate.yml
on: [pull_request]

jobs:
  refactoring_validation:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      
      - name: Detect Refactorings
        run: |
          java -jar RefactoringMiner.jar c1=${{ github.base_ref }} \
                                        c2=${{ github.head_ref }} \
                                        -json > refactorings.json
      
      - name: Run ReFD Danger Analysis
        run: |
          java -cp ReFD.jar ReFD \
               refactorings.json \
               src/ \
               --output=dangers.json
      
      - name: Check Dangers
        run: |
          # Fail if high-risk danger detected without double-review
          jq '.dangers[] | select(.severity=="HIGH")' dangers.json
          if [ $? -eq 0 ]; then
            echo "High-risk refactoring detected. Require code owner approval."
            exit 1
          fi
      
      - name: Test Coverage
        run: |
          mvn clean test jacoco:report
          COVERAGE=$(jq '.lineCoverage' target/jacoco.json)
          if (( $(echo "$COVERAGE < 75" | bc -l) )); then
            echo "Test coverage below 75%. Refactor rejected."
            exit 1
          fi
      
      - name: Static Analysis
        run: |
          sonar-scanner -Dsonar.projectKey=refactoring-audit
          # Extract SQALE debt delta
          curl -s https://sonarqube.company.com/api/measures/search_history \
               -d metricKeys=sqale_index | jq '.measures[].history[]'
```

### IX. Refactoring Patterns as Infrastructure Templates

**Pattern-Specific Execution Checklists**

**Pattern: Extract Method for Testability**

```checklist
[ ] Code location identified: method name, class, line range
[ ] Extraction scope validated: >3 statements, <15 LOC
[ ] ReFD analysis passed: No AddMethod dangers (double definition)
[ ] New method name review: Use verb phrase (getConfiguration, validateInput)
[ ] Access modifier assignment: Usually protected (for Mockito.spy override)
[ ] @VisibleForTesting annotation added (if package-private)
[ ] Call sites updated (within same class, no cross-module calls)
```

**Pattern: Extract Class for Dependency Isolation**

```checklist
[ ] Dependency identified: external service, database, mock-resistant
[ ] Class extraction scope: 3-7 related methods, <200 LOC
[ ] ReFD analysis: No MoveMethod local reference breaks
[ ] Constructor injection: Add parameter to parent class
[ ] Mock implementation: Create interface, test subclass
[ ] Cross-module references: Ensure extracted class doesn't violate layering
[ ] Test coverage: New class + integration with parent ≥85%
[ ] Cyclomatic complexity: Both parent and child CC <20
[ ] Behavior verification: Constructor + delegation calls preserved
[ ] Git commit: "REFACTOR: Extract PaymentGatewayClient from CheckoutService"
```

**Pattern: Widen Access for Assertion**

```checklist
[ ] Method/field identified: currently private, needed for test assertion
[ ] Reason documented: Why test cannot access through public interface
[ ] Alternative analysis: Can test be rewritten to avoid widening? (prefer)
[ ] Access level chosen: package-private or protected (prefer over public)
[ ] @VisibleForTesting annotation: Add with comment
[ ] No external dependency on widened member: Grep codebase for usage
[ ] Test assertion written: Verify narrowed access still enables assertion
[ ] Regression risk: <5% (widening is lowest-risk pattern)
[ ] Git commit: "REFACTOR: Widen access to validateCreditCard for testing"
```

### XI. Threat Model \& Failure Modes

**Failure Mode Analysis**


| Failure Mode | Cause | Detection | Mitigation |
| :-- | :-- | :-- | :-- |
| **Silent behavioral change** | LLM suggests unsafe refactoring; low test coverage | Regression test failure in CI; ReFD undetected danger | RefactoringMirror + 75% coverage gate |
| **Implicit contract break** | Refactored method has undocumented side effects | Integration test fails; downstream module breaks | Behavioral contracts + cross-module dependency analysis |
| **Performance regression** | Refactor introduces unnecessary loop or allocation | Benchmark comparison ±5% threshold breached | Performance test suite in CI |
| **Knowledge loss** | Developer makes refactoring without documenting reason | Code review catch; but long-term team understanding suffers | Mandatory @refactor: comments + changelog |
| **Over-refactoring** | Team refactors stable code for aesthetics, introduces churn | Code review pushback; no measured quality improvement | Define refactoring threshold (CC>15 only) |
| **Incomplete refactoring** | Tool-assisted refactor misses cross-file references | Compile error or runtime failure in integration test | Cross-module static analysis + canary release |
| **Cascade effect** | Refactor A enables refactor B; but A rollback breaks B | CI detects broken build; requires careful ordering | Atomic refactoring boundaries + sequential validation |

**Rollback Strategy**

```
// Refactoring commits include rollback hooks:
git tag refactor/v1.0-pre  // Pre-refactoring checkpoint
git commit -m "REFACTOR: Extract PaymentProcessor

  Rationale: Isolate payment logic for testability
  Pattern: Extract Class for Invocation
  Dangers checked: ReFD v2.0 (no HIGH risks)
  Rollback: git reset --hard refactor/v1.0-pre
  
  If regression detected:
  1. Notify #dev-platform
  2. Revert commit (1-click in GitHub UI)
  3. Root cause analysis in post-incident review"
```

### XIII. Constraints \& Success Criteria

**Hard Constraints (Must-Have)**

✅ **Behavioral Invariance**: Unit tests + integration tests must pass unchanged post-refactor
✅ **Semantic Delta Traceability**: Every refactoring diff must link to documented rationale + tracked smell/pattern
✅ **Test Coverage Non-Regression**: Δ coverage ≥ 0% (never allow test coverage to decrease)
✅ **Refactor Pattern Linkage**: Every refactoring PR must cite specific pattern from catalog (Extract Method, etc.)
✅ **High-Risk Audit Gate**: Auth, billing, payment modules require double-review + canary validation before merge

- Cyclomatic Complexity Improvement: Avg reduction ≥5% per refactored module
- Development Velocity: Maintain or increase (refactoring should not slow down feature work)

### XIV. Integration with Adjacent Ecosystems

**LangGraph Workflow for Autonomous Refactoring Agents**

```python
# Extended DRP pattern: AI-assisted refactoring agent
from langgraph.graph import StateGraph
from langchain_openai import ChatOpenAI
import subprocess

# State: Refactoring task + intermediate results
class RefactoringTask(TypedDict):
    code_context: str
    pattern: str  # "extract_method", "widen_access", etc.
    rationale: str
    refd_dangers: list[str]
    llm_suggestion: str
    refactoring_mirror_output: str
    validation_passed: bool

workflow = StateGraph(RefactoringTask)

# Node 1: Identify opportunities (LLM)
async def identify_opportunities(state):
    llm = ChatOpenAI(model="gpt-4-turbo")
    response = llm.invoke([
        {"role": "system", "content": "You are a refactoring expert..."},
        {"role": "user", "content": f"Analyze this code: {state['code_context']}..."}
    ])
    state['llm_suggestion'] = response.content
    return state

# Node 2: Danger analysis (ReFD)
async def run_refd_analysis(state):
    result = subprocess.run([
        "java", "-cp", "ReFD.jar", "ReFD",
        "--code", state['code_context'],
        "--pattern", state['pattern'],
        "--output", "json"
    ], capture_output=True, text=True)
    state['refd_dangers'] = json.loads(result.stdout)['dangers']
    return state

# Node 3: Validate with RefactoringMirror
async def validate_refactoring(state):
    # Extract refactorings from LLM output, reapply via IntelliJ
    result = subprocess.run([
        "java", "-cp", "RefactoringMirror.jar", "RefactoringMirror",
        "--original", state['code_context'],
        "--suggested", state['llm_suggestion'],
        "--engine", "intellij"
    ], capture_output=True, text=True)
    state['refactoring_mirror_output'] = json.loads(result.stdout)
    state['validation_passed'] = result.returncode == 0
    return state

# Node 4: Human review gate
async def human_review_required(state):
    # If high-risk danger detected, require human review
    high_risk = any(d['severity'] == 'HIGH' for d in state['refd_dangers'])
    return 'human_review' if high_risk else 'merge'

# Build graph
workflow.add_node("identify_opportunities", identify_opportunities)
workflow.add_node("run_refd_analysis", run_refd_analysis)
workflow.add_node("validate_refactoring", validate_refactoring)
workflow.add_conditional_edges(
    "validate_refactoring",
    human_review_required,
    {"human_review": "human_review", "merge": "merge"}
)

app = workflow.compile()
```

### Conclusion

Refactoring transformed from an optional practice into a critical infrastructure capability requires three simultaneous shifts:

1. **From Craft to Systematic**: Replace intuition-driven refactoring with empirically-validated patterns, formal risk models (ReFD), and measurable success criteria (SQALE, CRAP).

3. **From Local to Organizational**: Instrument refactoring as observable, auditable infrastructure—tracked in version control, validated by CI/CD gates, visualized in dashboards, governed by clear decision frameworks.

**The Framework's Core Promise**: Enable teams to refactor fearlessly, knowing that every change is semantically validated, every danger is diagnosed before execution, and every improvement is measurable. This unlocks the latent productivity locked in ossified codebases.
