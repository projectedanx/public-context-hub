## Part I: Topological Map of the Marketplace

The GitHub Actions Marketplace is conventionally indexed by marketing function: Deployment, Security, Code Quality, Notifications. This taxonomy is useful for discoverability but catastrophically misleading for architectural synthesis. It obscures the *geometric operation* each action performs on the repository's state graph. The following re-classification resolves this.[^1]

### Geometric Operation Taxonomy

A more structurally faithful taxonomy organizes actions by the *type of transformation* they apply to the repository's state manifold. Five primitive geometric types emerge from this analysis:

**Type I — Projectors:** Actions that reshape or reduce state dimensionality. They take a high-entropy input (raw code, diff, log stream) and output a lower-dimensional structured signal. Examples: `super-linter` (projects source code → violation vector), `TruffleHog OSS` (projects full repo → credential signal), `gosec` (projects Go source → security signal). The defining property is *information compression with semantic preservation*.[^2]

**Type II — Traversals:** Actions that iterate over a code graph or dependency DAG and emit ordered signals. Examples: `actions/checkout` (traverses the commit DAG to a target ref), `actions-rust-cross` (traverses a build target matrix), semantic-release variants (traverse the commit log for conventional commit patterns). The defining property is *ordered iteration with stateful accumulation*.[^3]

**Type III — Escrows:** Actions that hold state invariant until a condition is externally satisfied. Examples: `actions/cache` (holds dependency state across runs), `actions/upload-artifact` + `actions/download-artifact` (holds inter-job state across runner isolation boundaries), environment protection rules with required reviewers (holds deployment state pending human signal). The defining property is *conditional temporal invariance*.[^4]

**Type IV — Oracles:** Actions that query an external system and inject truth into the workflow's decision graph. Examples: `github-script` (queries the GitHub REST API), LLM code review actions (queries an inference endpoint), `run-digger` (queries Terraform state from remote backend). The defining property is *external truth injection into the local decision graph*.[^5]

**Type V — Synthesizers:** Actions that generate new code, configuration, or workflow topology from runtime signals. Examples: `opencommit` (synthesizes commit messages from diffs), LLM PR description generators, and — critically — actions that use `octokit` to write new `.github/workflows/*.yml` files via the Contents API. The defining property is *generation of new executable topology from existing signals*.[^6][^7]

### The Negative Space

The highest-value architectural territory lies not within these categories but *between them* — specifically at these interfaces:
            - **Projector × Oracle:** A security scanner (Projector) whose compressed output is fed to an LLM Oracle that interprets semantic meaning and writes a remediation plan
            - **Escrow × Synthesizer:** Artifact state held over time that, when a condition fires, triggers a Synthesizer to generate new workflow YAML
            - **Traversal × Oracle:** A matrix traversal that fans out over discovered microservices and queries each via Oracle actions for live health data
            - **Projector × Escrow × Synthesizer:** The full autopoietic loop — project state, hold it against a temporal invariant, synthesize new code to resolve the delta

This negative space is where the five novel architectures below live.

## Part II: The Five Novel Workflow Architectures

### Architecture 1: The Recursive Autopoietic Security Loop (RASL)

**Geometric Type Composition:** Projector(Security) × Oracle(LLM) × Synthesizer(PR-Creator) × Traversal(Scheduler)
**Pattern Class:** Recursive Autopoiesis Hypothesis — a self-improving, zero-human-intervention security patch loop

**Conceptual Architecture:**
A `schedule` trigger fires `gosec` (Projector) and `TruffleHog` (Projector) in parallel. Their JSON outputs are merged and forwarded to an LLM Oracle action (calling Claude 4.6 Opus or GPT-5.4 via their APIs) with a strict schema-constrained prompt requesting a JSON remediation plan containing: `file_path`, `line_number`, `patch_content`, and `confidence_score`. The Synthesizer job reads this plan, generates the patch files using a Python script, commits them to a branch, and opens a PR via `github-script` (Oracle). A post-synthesis job re-runs only the affected scanners against the patch branch as a negative control — if violations persist, the PR is labeled `needs-human-review`; if resolved, it is auto-merged via the API.[^6][^5]

The key non-obvious mechanism: the LLM Oracle is not asked to "fix the code" (which triggers Semantic Saponification). It is asked to produce a *structured JSON patch plan* with a confidence threshold — the `EpistemicEscrow` analog. Patches below `cfd_threshold=0.15` confidence halt synthesis and escalate. This prevents hallucinated remediation from corrupting the codebase.

          import json, os

                  try: findings[f] = json.load(fh)
                  except: findings[f] = []
          with open('merged-findings.json', 'w') as out:
              json.dump(findings, out)

          import json, os, urllib.request, urllib.error

          with open('merged-findings.json') as f:
              findings = json.load(f)

          # Structured schema-constrained prompt — prevents Semantic Saponification
          system_prompt = """You are a security patch planner. 
          Given security scan findings, return ONLY valid JSON matching this schema:
          {
            "patches": [
              {
                "file_path": "string",
                "issue_type": "string", 
                "patch_description": "string",
                "confidence": 0.0-1.0,
                "patch_content": "string (unified diff format)",

              }

              "model": "claude-opus-4-6",
              "max_tokens": 4096,

              "https://api.anthropic.com/v1/messages",

    runs-on: ubuntu-latest
    needs: llm-remediation-planner
    if: needs.llm-remediation-planner.outputs.should-synthesize == 'true'
    steps:
      - uses: actions/checkout@v4
      - uses: actions/download-artifact@v4

          name: llm-patch-plan

### Architecture 2: The Distributed Asynchronous State Machine (DASM)

**Geometric Type Composition:** Escrow(Artifact) × Traversal(Matrix) × Oracle(State-Reader)
**Pattern Class:** Asynchronous State-Passer — exploiting artifact I/O as a faux message-passing bus

**Conceptual Architecture:**
This architecture inverts the conventional assumption that GitHub Actions jobs are stateless. By treating `actions/upload-artifact` and `actions/download-artifact` as message queue analogues, a 10-node matrix workflow can implement a distributed state machine where each node reads prior state, applies a transition function, and writes new state to a named artifact slot. The matrix strategy generates job IDs dynamically from a prior job's output, enabling a runtime-defined state graph.[^8][^9][^4]

The non-obvious mechanism: the artifact *name* encodes the state transition. By naming artifacts `state-transition-{from}-{to}-{run_id}`, a final aggregation job can reconstruct the full state transition matrix and verify DAG coherence. This is structurally isomorphic to an actor model message bus running over blob storage.

```yaml
# Architecture 2: Distributed Asynchronous State Machine (DASM)
# Pattern: Artifact-as-message-queue, dynamic matrix as distributed node set
name: DASM — Distributed State Machine via Artifact Bus

on:
  push:
    branches: [main]
  workflow_dispatch:
    inputs:
      node_count:
        description: 'Number of state machine nodes (max 10)'
        default: '5'
        type: string

jobs:
  # === PHASE 1: TRAVERSAL — Define state graph dynamically ===
  define-state-graph:
    name: Traversal — Generate State Node Matrix
    runs-on: ubuntu-latest
    outputs:
      nodes: ${{ steps.gen.outputs.nodes }}
      initial-state-artifact: initial-state
    steps:
      - uses: actions/checkout@v4
      - name: Generate state node configurations
        id: gen
        run: |
          python3 << 'EOF'
          import json, os
          node_count = int(os.environ.get('NODE_COUNT', '5'))
          # State graph: each node has id, transition_fn, input_state_key, output_state_key
          nodes = [
              {
                  "id": f"node_{i:02d}",
                  "transition_fn": ["validate", "transform", "enrich", "compress", "verify"][i % 5],
                  "input_key": f"state_{i:02d}",
                  "output_key": f"state_{(i+1):02d}"
              }
              for i in range(node_count)
          ]
          print(f"nodes={json.dumps(nodes)}")
          EOF
          >> "$GITHUB_OUTPUT"
        env:
          NODE_COUNT: ${{ github.event.inputs.node_count || '5' }}

      - name: Initialize state (Escrow write)
        run: |
          echo '{"version": 1, "data": {}, "transitions": [], "coherent": true}' > initial-state.json
      - uses: actions/upload-artifact@v4
        with:
          name: state_00
          path: initial-state.json
          retention-days: 1

  # === PHASE 2: ESCROW NODES — Each processes and passes state ===
  state-node:
    name: Escrow Node — ${{ matrix.node.id }} (${{ matrix.node.transition_fn }})
    runs-on: ubuntu-latest
    needs: define-state-graph
    strategy:
      matrix:
        node: ${{ fromJson(needs.define-state-graph.outputs.nodes) }}
      max-parallel: 1  # Sequential state passing (linearized DAG)
    steps:
      - uses: actions/checkout@v4
      - name: Read prior state (Escrow read)
        uses: actions/download-artifact@v4
        with:
          name: ${{ matrix.node.input_key }}
          path: ./state-in/
      - name: Apply state transition function
        run: |
          python3 << 'EOF'
          import json, os, hashlib, time

          node_id = "${{ matrix.node.id }}"
          transition_fn = "${{ matrix.node.transition_fn }}"

### Architecture 3: The Polyglot Test Resonator (PTR)

**Geometric Type Composition:** Projector(Rust-Fuzzer) × Traversal(AST-Walker) × Oracle(Cross-Language Validator)
**Pattern Class:** Epistemic Translation — mapping formal verification semantics across language paradigms

**Conceptual Architecture:**
A Rust-based property testing action (`cargo-fuzz` or `quickcheck`) generates a corpus of edge-case inputs against a Rust function specification. These inputs are serialized to JSON and fed via artifact to a TypeScript or Python job that ingests them as property-test seeds for its own native fuzz framework (`fast-check` for TypeScript, `hypothesis` for Python). The *structural claim* is that valid edge cases in a strongly-typed Rust specification expose behavioral divergences in dynamically-typed equivalents — cross-language behavioral resonance.[^10][^11][^12]

### Architecture 4: The Temporal Escrow Smart Contract (TESC)

**Geometric Type Composition:** Escrow(Artifact-Hash) × Oracle(External-Condition) × Synthesizer(Conditional-Deploy)
**Pattern Class:** Temporal Escrow Hypothesis — GitHub as decentralized conditional execution environment

**Conceptual Architecture:**
This architecture implements the Temporal Escrow Hypothesis: a deployment that is held mathematically invariant until an external, verifiable condition is satisfied — turning a GitHub repository into a smart-contract-like execution environment. A cron job hashes the target deployment artifact and stores it with a timestamp. A separate polling job checks an external condition (API endpoint health, market signal, or stakeholder approval token) and only releases the deployment when condition AND hash invariance are both satisfied.[^13][^14]

```yaml

    inputs:

  ESCROW_ARTIFACT: deployment-escrow

    if: github.event_name == 'push' || github.event.inputs.action == 'escrow'

      escrow-timestamp: ${{ steps.hash.outputs.timestamp }}
    steps:
      - uses: actions/checkout@v4
      - name: Build/collect deployment artifact
        run: |
          # Simulate build output — replace with actual build command
          mkdir -p dist
          echo "deployment-payload-$(git rev-parse --short HEAD)" > dist/app.tar.gz
      - name: Hash and seal escrow
        id: hash
        run: |
          HASH=$(sha256sum dist/app.tar.gz | awk '{print $1}')
          TS=$(date -u +%Y%m%dT%H%M%SZ)
          echo "artifact_hash=$HASH" >> "$GITHUB_OUTPUT"
          echo "timestamp=$TS" >> "$GITHUB_OUTPUT"
          python3 -c "
          import json
          escrow = {
              'hash': '$HASH',
              'timestamp': '$TS',
              'commit': '$(git rev-parse HEAD)',
              'branch': '$(git branch --show-current)',
              'status': 'ESCROWED',
              'condition_met': False,
              'release_authorized': False

          "
      - uses: actions/upload-artifact@v4
        with:
          name: ${{ env.ESCROW_ARTIFACT }}
          path: |
            dist/
            escrow-manifest.json
          retention-days: 30  # Temporal escrow window

  condition-oracle:
    name: Oracle — External Condition Poller
    runs-on: ubuntu-latest
    if: github.event_name == 'schedule' || github.event.inputs.action == 'check'
    outputs:
      condition-met: ${{ steps.check.outputs.condition_met }}
      condition-signal: ${{ steps.check.outputs.signal }}
    steps:
      - name: Query external condition
        id: check
        run: |
          python3 << 'EOF'
          import urllib.request, json, os, sys

          endpoint = os.environ.get('CONDITION_ENDPOINT', '')
          
          if not endpoint:
              # Fallback: check if a specific GitHub environment is approved
              # In production: query external service, smart contract, or approval API
              print("condition_met=false")
              print("signal=NO_ENDPOINT_CONFIGURED")
              with open(os.environ['GITHUB_OUTPUT'], 'a') as f:
                  f.write("condition_met=false\n")
                  f.write("signal=NO_ENDPOINT_CONFIGURED\n")
          else:
              try:
                  req = urllib.request.Request(endpoint, headers={"Accept": "application/json"})
                  with urllib.request.urlopen(req, timeout=10) as resp:
                      data = json.loads(resp.read())
                      met = data.get('release_authorized', False)
                      signal = data.get('signal', 'UNKNOWN')
                      with open(os.environ['GITHUB_OUTPUT'], 'a') as f:
                          f.write(f"condition_met={'true' if met else 'false'}\n")
                          f.write(f"signal={signal}\n")
              except Exception as e:
                  with open(os.environ['GITHUB_OUTPUT'], 'a') as f:
                      f.write("condition_met=false\n")
                      f.write(f"signal=ERROR_{str(e)[:50]}\n")
          EOF
        env:
          CONDITION_ENDPOINT: ${{ env.CONDITION_ENDPOINT }}

  # === PHASE 3: SYNTHESIZER — Conditional release + hash verification ===
  conditional-release:
    name: Synthesizer — Escrow Release (Hash-Verified)
    runs-on: ubuntu-latest
    needs: condition-oracle
    if: needs.condition-oracle.outputs.condition-met == 'true'
    environment: production  # Requires environment protection rules

```

Hash invariance is over the artifact file, not its semantic content — a malicious actor with write access to the `dist/` branch could substitute an artifact pre-escrow.[^13]

### Architecture 5: The Metacognitive Repository Brain (MRB)

**Geometric Type Composition:** Oracle(LLM-Analyst) × Synthesizer(Workflow-Writer) × Traversal(Issue-Parser) × Projector(Code-Summarizer)
**Pattern Class:** Sovereign Cognitive Operating System — the repository observes, reasons about, and restructures its own operational topology

**Conceptual Architecture:**
This is the most structurally radical architecture — it implements the Recursive Autopoiesis Hypothesis at the *workflow topology level* rather than the code level. A weekly trigger causes an LLM Oracle to analyze the repository's own `.github/workflows/` files, open issues, and PR merge history. It then synthesizes a *new or modified workflow file* that adapts the CI/CD topology to the observed patterns — e.g., if most PRs fail on Python lint, it adds a pre-commit lint gate; if build times are trending up, it restructures the job DAG for better parallelism.[^15][^14][^16][^5]

```yaml

              owner: context.repo.owner,
              repo: context.repo.repo,
              per_page: 50,
              status: 'completed'
            });

            // Collect recent issues (CI-related)
            const issues = await github.rest.issues.listForRepo({
              owner: context.repo.owner,
              repo: context.repo.repo,
              state: 'open',
              per_page: 20,
              labels: 'ci,bug,performance'
            });

            // Collect existing workflow files
            let workflows = [];
            try {
              const wf_dir = await github.rest.repos.getContent({
                owner: context.repo.owner,
                repo: context.repo.repo,
                path: '.github/workflows'
              });
              workflows = wf_dir.data.map(f => ({ name: f.name, sha: f.sha, size: f.size }));
            } catch(e) { workflows = []; }

            // Compute workflow failure rates
            const failure_rates = {};
            for (const run of runs.data.workflow_runs) {
              const key = run.name || 'unknown';
              if (!failure_rates[key]) failure_rates[key] = { total: 0, failed: 0 };
              failure_rates[key].total++;
              if (['failure', 'timed_out'].includes(run.conclusion)) {
                failure_rates[key].failed++;
              }
            }

            const signals = {
              workflow_failure_rates: Object.fromEntries(
                Object.entries(failure_rates).map(([k, v]) => [k, {
                  rate: v.failed / Math.max(1, v.total),
                  total_runs: v.total

              ),
              open_ci_issues: issues.data.map(i => ({ title: i.title, labels: i.labels.map(l => l.name) })),
              existing_workflows: workflows,
              repo_context: {
                name: context.repo.repo,
                owner: context.repo.owner,
                timestamp: new Date().toISOString()
              }
            };

            console.log('Signals aggregated:', JSON.stringify(Object.keys(signals)));

      - uses: actions/upload-artifact@v4
        with:
          name: repo-signals
          path: repo-signals.json
          retention-days: 1

  llm-topology-analyst:
    name: Oracle — LLM CI/CD Topology Analyst
    runs-on: ubuntu-latest

    outputs:
      recommendation-artifact: topology-recommendation
      has-recommendations: ${{ steps.analyze.outputs.has_recs }}

      - uses: actions/download-artifact@v4

              for f in os.listdir(wf_dir):
                  if f.endswith('.yml') or f.endswith('.yaml'):
                      with open(os.path.join(wf_dir, f)) as fh:
                          wf_contents[f] = fh.read()[:3000]  # First 3000 chars
          with open('workflow-contents.json', 'w') as out:

Self-referential modification of the MRB workflow file itself is prohibited by a `paths-ignore` rule on the triggering workflow.

## Part IV: Reflexive Check \& Relational Inclusions

### Identified Blind Spots

**Bit-Rot in Niche Actions:** The DRP specifically warned against treating star count as structural utility proxy. Actions like `gosec` and `TruffleHog` are actively maintained, but any niche action in a cross-category composition (e.g., a markdown link checker repurposed as an API route validator) may silently fail or change its output schema. The mitigation is SHA-pinned action references and integration tests that verify output schema, not just exit codes.[^2]

**Runner Environment Undocumented Limits:** GitHub's runner sandbox imposes undocumented limits on artifact I/O throughput that can affect the DASM architecture. Empirical measurement during staging is required before production adoption.[^1]

**LLM Non-Determinism as Security Surface:** The AIDev study covering 932,791 agentic PRs across five AI coding agents (Copilot, Devin, Cursor, Claude Code, Codex) found that LLM-generated PRs introduce systematic bias patterns — specifically, agents tend to produce over-confident patches for well-known CWEs while underperforming on novel vulnerability classes. RASL and MRB must implement output schema validation as a non-negotiable gate.[^16]

### Relational Inclusions — Cross-Ecosystem Translation

**GitLab CI Equivalents:**
            - DASM's artifact bus → GitLab's `artifacts:reports` with `dependencies:` keyword
            - TESC's escrow → GitLab Environments with protected branches + manual `when: manual` gates
            - MRB → GitLab's Auto DevOps with AI-powered pipeline configuration

**Model Context Protocol (MCP) Boundary:**
The architectures above operate within the GitHub Actions execution boundary. The MCP boundary begins where the action *exits the runner and calls a local tool*. For RASL and MRB, the LLM Oracle step sits precisely at this boundary — the action makes an HTTP call to an inference endpoint, which is equivalent to an MCP tool invocation. Future extensions should replace the raw API call with an MCP-compliant tool definition, enabling the IDE (Cursor, VS Code Copilot) to participate in the same cognitive loop as the CI/CD pipeline.

**Bitbucket Pipelines Translation:**
            - Artifact bus → Bitbucket Pipelines `artifacts:` key with `shared: true`
            - Matrix strategy → `parallel:` step with dynamic configuration via pipe outputs
            - Oracle pattern → Bitbucket Pipes (the marketplace equivalent) wrapping external API calls


### Falsification Summary

The five architectures are collectively falsified if:

1. GitHub introduces strict cross-job artifact isolation preventing the DASM message-bus pattern
2. Anthropic/OpenAI API non-determinism rates exceed 15% for schema-constrained outputs, making RASL and MRB unreliable
3. The GHA YAML schema parser rejects dynamically composed workflow files written via the Contents API in the MRB loop
4. Runner minute quotas at the free tier make TESC's polling interval economically nonviable

## Part V: Validation Protocol

**Positive Tests:**
            - Run DASM with `node_count=5` and verify the final state's `transitions` array contains exactly 5 entries with correct `transition_fn` values[^8][^4]
            - Run PTR with a corpus including `float_edge: Infinity` and verify TypeScript reports a divergence (since `JSON.stringify(Infinity) === 'null'` in JavaScript)
            - Run RASL with a deliberately seeded vulnerability (e.g., a hardcoded API key in a test file) and verify the full loop completes: scan → plan → patch PR

**Negative Controls (DCCDSchemaGuard Intercepts):**
            - Submit a workflow with a circular dependency (`job_a needs job_b`, `job_b needs job_a`) — GitHub's parser must reject with a schema violation
            - Submit YAML with an invalid `on:` trigger key — must fail validation before runner invocation
            - Attempt to reference an artifact from a *previous workflow run* (not the current run) in DASM — verify the download step fails with a `not found` error, confirming the artifact namespace is run-scoped

**The Projection Tax (Quantitative):**
Each architecture incurs a "Projection Tax" — the information loss inherent in reducing a high-dimensional input to a structured signal. For RASL, this tax is approximately 35-60% of vulnerability context is dropped between raw scan output and the LLM's JSON patch plan. This is not a defect; it is the expected cost of compression. The metric to track is whether the *dropped information contains false negatives* — vulnerabilities the LLM chose not to patch due to complexity. Monitoring this requires logging the full LLM response, including any patches the confidence filter rejected, for manual quarterly review.[^14][^15]
