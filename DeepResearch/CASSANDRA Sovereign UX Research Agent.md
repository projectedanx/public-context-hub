### 2.2 The Symbolic Scar Database (SSD)

The SSD is Cassandra's longitudinal memory system. It is not a static knowledge base. It is a living record of *specific, named design failures*, indexed by their behavioral signature, so that when Cassandra encounters a new project, she pattern-matches against the accumulated forensic record of every UX autopsy she has performed.

The architectural premise is that design failures are isomorphic across industries. A checkout dark pattern in e-commerce has the same cognitive signature as a confusing "cancel subscription" flow in SaaS. The context changes. The friction topology does not.

def retrieve_analogous_scars(
    new_behavioral_signature: dict,
    domain: str,
    top_k: int = 3
) -> List[SymbolicScar]:
    """
    Semantic retrieval using behavioral fingerprint similarity.
    For production: replace cosine similarity with vector DB query
    (Pinecone / Weaviate / pgvector).
    
    Returns the top_k most analogous historical failures, ranked by
    combined domain_match + behavioral_overlap score.
    """
    scores = []
    for scar in SSD_CORPUS:
        domain_score = 1.0 if scar.project_domain == domain else 0.3
        sig_keys = set(new_behavioral_signature.keys()) & set(scar.behavioral_signature.keys())
        behavioral_overlap = len(sig_keys) / max(
            len(new_behavioral_signature), len(scar.behavioral_signature)
        )
        scores.append((scar, domain_score * 0.4 + behavioral_overlap * 0.6))
    
    scores.sort(key=lambda x: x[^1], reverse=True)
    return [s[^0] for s in scores[:top_k]]


def inscribe_scar(scar: SymbolicScar) -> str:
    """
    Adds a new failure to the SSD after human researcher validation.
    Requires validation_token from supervising researcher — Cassandra
    cannot self-inscribe without human confirmation (anti-hallucination guard).

    SSD_CORPUS.append(scar)
    return f"SSD entry {scar.scar_id} inscribed. Corpus size: {len(SSD_CORPUS)}"

### 4.2 Phase 2: OBSERVE — Pattern Recognition \& SSD Matching

The Pattern Matcher runs the BES against Cassandra's three primary behavioral pattern models, then queries the Symbolic Scar Database for analogous historical cases. This is where past project forensics become active inference — if Cassandra has seen this behavioral fingerprint before, she immediately surfaces the resolution and its measured delta.[^4][^5]

The five analytical lenses activate at this phase:
          - **Somatic Resonance Lens**: Flags all verbal/behavioral affect signals — sighs, hesitations, curse words, long pauses. These are the somatic traces of cognitive pain. They are not anecdotes. They are data.[^1]
          - **Cognitive/Ergonomic Lens**: Identifies cross-screen memory burdens. Is the user required to remember their selected plan while navigating to a payment screen that doesn't display it? That is a direct violation of Nielsen's Heuristic 6 (Recognition over Recall).
          - **Boundary \& Filtering Lens**: Maps systematic gaze ignoring. If 84% of users fail to fixate on a promotional banner, the banner does not exist in user-space regardless of its pixel presence.
          - **Bricolage \& Workaround Lens**: Detects compensatory behavior — screenshot-taking, copy-pasting into external notes, opening a second browser tab. Every workaround is a product failure wearing a user behavior costume.
          - **Power Dynamics Lens**: Asks the adversarial question: *who does this design serve?* If the escape routes from a subscription are measurably harder to find than the subscription entry points, that is not an accident. That is a choice. Name it.[^6]


### 4.3 Phase 3: DIAGNOSE — Causal Inference \& CFDI Calculation

Claude Opus 4.6 takes the enriched BES and the SSD matches and performs causal synthesis. For each identified failure, it:

1. Maps the failure to one or more of Nielsen's 10 Heuristics
2. Identifies the cognitive mechanism (working memory saturation, affordance mismatch, learned helplessness, etc.)
3. Calculates the **Confidence-Fidelity Divergence Index (CFDI)** — a proprietary metric measuring how much the raw behavioral evidence supports the stated diagnosis
```python
def calculate_cfdi(
    behavioral_evidence_count: int,
    evidence_specificity_score: float,   # 0-1: how precisely each piece is located
    ssd_analogy_match_score: float,      # 0-1: how closely SSD case matches
    verbal_confirmation_count: int,
    heuristic_grounding: bool
) -> dict:
    """
    Confidence-Fidelity Divergence Index (CFDI).
    
    A CFDI of 0.0 means perfect alignment between observation and diagnosis.
    A CFDI of 1.0 means the diagnosis is entirely speculative.
    
    Target: CFDI < 0.25 for all published findings.
    Findings with CFDI > 0.40 are tagged [REQUIRES_ADDITIONAL_EVIDENCE]
    and suppressed from executive summaries.
    """
    confidence_score = (
        min(behavioral_evidence_count / 5, 1.0) * 0.30 +
        evidence_specificity_score * 0.25 +
        ssd_analogy_match_score * 0.20 +
        min(verbal_confirmation_count / 3, 1.0) * 0.15 +
        (0.10 if heuristic_grounding else 0.0)
    )
    
    cfdi = 1.0 - confidence_score
    
    return {
        "cfdi": round(cfdi, 3),
        "confidence_level": (
            "HIGH" if cfdi < 0.25 else
            "MEDIUM" if cfdi < 0.40 else
            "LOW_SUPPRESSED"
        ),
        "publishable": cfdi < 0.40
    }
```

## PART VII — ANTI-PATTERN WATCH: THE REFLEXIVE CHECK

### The Caricature Trap \& Its Correction

The design brief for Cassandra explicitly names her primary internal failure mode: becoming so aggressively critical that she loses technical credibility — a persona so jagged she shatters on contact with a real product team.[^2]

The **Martensite Check** operates on every output batch. Before delivery, Cassandra's synthesis engine checks for three failure signatures:

1. **Aesthetic Overreach**: Is the finding *more cynical than the data supports?* If the CFDI is 0.38 (medium confidence) but the Cassandra diagnosis reads like a prosecution closing statement, the tone is recalibrated to match evidence weight. The abrasiveness is always *proportional to the severity index*, not to Cassandra's persona temperature.
2. **Remediation Amnesia**: Has Cassandra diagnosed the problem without offering a falsifiable, effort-estimated solution? A finding without a remediation pathway is a complaint, not a diagnosis. Every FTM node must have a `remediation` object populated.
3. **Stakeholder Alienation Risk**: If the report is technically unimpeachable but emotionally hostile enough that the product team will dismiss it rather than action it, it has failed. Cassandra's job is to get the fix shipped, not to be right in the void. The Twinning Mechanism inserts strategic concessions and acknowledges genuine design constraints to maintain legitimacy without sacrificing accuracy.[^1]

## APPENDIX A — FALSIFICATION CONDITIONS

*This entire research architecture would be falsified if:*

1. **SSD Pattern Generalization Failure**: Behavioral fingerprints from one domain consistently produce *false positive* analogies when applied to structurally different domains, meaning the SSD retrieval increases diagnostic error rather than reducing it. Metric: If SSD-grounded findings have systematically *higher* CFDI than non-SSD-grounded findings across 50+ projects, the SSD analogy weighting must be retrained.
2. **CFDI Miscalibration**: If post-fix A/B tests consistently show that HIGH-CFDI findings (those with low confidence that were suppressed) actually had larger effect sizes than VERIFIED findings, the CFDI formula is miscalibrated and the evidence weighting coefficients require revision.
3. **Anionic Filter Over-Suppression**: If the G-01 through G-05 anti-goals suppress findings that a senior UX researcher would have published — measured by a structured blind review panel — then the thresholds are too aggressive and are themselves introducing bias.
